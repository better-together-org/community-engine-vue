# PWA Strategy for Community Engine Vue

**Status:** Design complete ‚Äî implementation in `src/pwa/`
**Scope:** This document covers the PWA layer as it relates to CEV (the plugin library). The actual service worker registration lives in the host app (e.g. `better-together-vue`).

---

## Guiding Principle

**PGlite is the source of truth locally; the server is a sync target, not a prerequisite.**

Everything that can work offline does work offline. The PWA layer adds:
1. Reliable asset caching so the app shell loads instantly from cache
2. Background sync so queued mutations are replayed even after the tab closes
3. Install prompt support for the "Add to Home Screen" experience
4. Silent SW updates with a user-controlled reload banner

---

## 1. Toolchain

| Tool | Role |
|------|------|
| `vite-plugin-pwa` | SW generation, precaching manifest, update lifecycle |
| Workbox | Runtime caching strategies, BackgroundSync plugin |
| CEV `src/pwa/index.js` | Composables + Workbox config helper for host apps |
| CEV `src/pwa/sw-helpers.js` | SW-context route registration for background sync |

**CEV is a library** ‚Äî `vite-plugin-pwa` is NOT installed in CEV. Host apps install it and use `getCevWorkboxConfig()` to populate `workbox.runtimeCaching`.

---

## 2. Service Worker Registration

```
registerType: 'autoUpdate'
```

- The SW is updated silently in the background.
- A new SW version activates on the next navigation (tab close + reopen, or programmatic `skipWaiting`).
- When an update is detected, the SW posts a message to the page which CEV's `useSwUpdate()` composable surfaces as `updateAvailable = true`.
- The host app optionally shows a "New version available ‚Äî reload" banner; `applyUpdate()` calls `skipWaiting` then reloads.

**Scope:** The host app registers the SW at `/` (default). CEV does not register its own SW.

## 3. Cache Strategies by Resource Type

| Resource | Strategy | Cache Name | TTL / Limits | Notes |
|----------|----------|------------|--------------|-------|
| App shell (HTML, CSS, JS, icons) | **CacheFirst** (precache) | `workbox-precache-*` | Versioned ‚Äî invalidated on deploy | Generated by `vite-plugin-pwa`'s `generateSW` via `globPatterns` |
| CE Rails JSON:API (`/bt/api/*`) | **NetworkFirst** | `ce-api-cache` | 24h, network timeout 5s | Auth routes (`/bt/api/auth/*`) are excluded ‚Äî see ¬ß7 |
| Images / user uploads | **CacheFirst** | `ce-image-cache` | 100 entries, 7 days | Covers `png jpg jpeg svg gif webp` extensions |
| Web fonts | **StaleWhileRevalidate** | `ce-font-cache` | 30 entries, 365 days | Serve cached immediately, refresh in background |

### NetworkFirst timeout behaviour

For API responses: if the network does not respond within 5 seconds, Workbox falls back to the cached response. This ensures the app remains usable on flaky connections without long blank states.

## 4. Background Sync Integration

### Without service worker (current behaviour)

`useSyncStore.drainSyncQueue()` runs when:
- The tab regains network connectivity (`online` event via `initNetworkListeners()`)
- The host app calls `drainSyncQueue()` manually

This works while the tab is open.

### With service worker (PWA layer)

`registerBackgroundSyncRoutes(workbox)` (from `src/pwa/sw-helpers.js`) adds a Workbox `BackgroundSyncPlugin` on `POST / PATCH / DELETE` to `/bt/api/*`.

When a mutation request fails while offline:
1. Workbox queues it in `ce-api-mutation-queue` (IndexedDB, managed by Workbox).
2. When the browser reconnects ‚Äî **even if the tab is closed** ‚Äî the browser wakes the SW and replays the queue.
3. On successful replay, the SW broadcasts a `ce-bg-sync-replayed` message to all open tabs.
4. If the tab is open, `useSyncStore` can listen for this message and call `drainSyncQueue()` to refresh the pending count.

### Coordination between layers

```
Tab open + online  ‚Üí  drainSyncQueue()  (useSyncStore)
Tab closed / SW wakes  ‚Üí  BackgroundSyncPlugin  (Workbox)
```

Both layers are additive ‚Äî they do not conflict. The BackgroundSyncPlugin only replays requests that failed (network error), not requests that succeeded. `drainSyncQueue()` counts rows in PGlite, not the Workbox queue.

### Wire-up in host app SW

```js
// src/sw.js (host app custom service worker)
import { precacheAndRoute } from 'workbox-precaching'
import { registerRoute } from 'workbox-routing'
import { NetworkOnly } from 'workbox-strategies'
import { BackgroundSyncPlugin } from 'workbox-background-sync'
import { registerBackgroundSyncRoutes } from '@bettertogether/community-engine-vue/pwa/sw-helpers'

precacheAndRoute(self.__WB_MANIFEST)

registerBackgroundSyncRoutes({ registerRoute, NetworkOnly, BackgroundSyncPlugin })
```

## 5. Install Prompt UX

CEV exports `useInstallPrompt()` ‚Äî host apps decide how and when to show install UI.

```js
const { canInstall, install, dismiss } = useInstallPrompt()
```

| Property | Type | Description |
|----------|------|-------------|
| `canInstall` | `Ref<boolean>` | `true` when browser has deferred a prompt and user hasn't dismissed recently |
| `install()` | `async () => { outcome }` | Triggers native browser install prompt |
| `dismiss()` | `() => void` | Suppresses prompt for 30 days (localStorage) |

**Dismiss persistence:** Stored in `localStorage` under `cev_install_prompt_dismissed_until` as a Unix timestamp. Checked on every `beforeinstallprompt` event.

**Environment safety:** All `window` / `localStorage` access is guarded with `typeof window !== 'undefined'` so the composable is safe in SSR/test environments.

## 6. Update Banner Flow

```
New SW waiting
    ‚îÇ
    ‚ñº
SW broadcasts message ‚Üí window dispatches `sw-updated` CustomEvent
    ‚îÇ
    ‚ñº
useSwUpdate() sets updateAvailable = true
    ‚îÇ
    ‚ñº
Host app shows "New version available ‚Äî reload" banner
    ‚îÇ
    ‚ñº
User clicks reload ‚Üí applyUpdate() ‚Üí SW.postMessage(SKIP_WAITING) ‚Üí location.reload()
```

CEV emits `sw-updated` as a standard DOM `CustomEvent` with `detail.registration` set to the `ServiceWorkerRegistration`. This decouples CEV from any specific vite-plugin-pwa version.

When using vite-plugin-pwa's built-in update detection, wire it up in `vite.config.js`:

```js
VitePWA({
  registerType: 'autoUpdate',
  devOptions: { enabled: true },
  // vite-plugin-pwa calls this when a new SW is waiting:
  // use dispatchSwUpdatedEvent from sw-helpers (runs in app context, not SW)
})
```

Or use vite-plugin-pwa's `useRegisterSW` hook alongside `useSwUpdate()`:

```js
import { useRegisterSW } from 'virtual:pwa-register/vue'
import { useSwUpdate } from '@bettertogether/community-engine-vue'

// useRegisterSW handles vite-plugin-pwa's specific lifecycle;
// useSwUpdate handles the generic sw-updated event for manual SW setups.
```

## 7. What Is NOT Cached

| Resource | Reason |
|----------|--------|
| `/bt/api/auth/*` | Auth tokens must never be served from cache ‚Äî always network-only |
| Auth tokens (Pinia state) | Stored in memory only, never serialised to cache storage |
| PGlite data (`idb://community-engine`) | Lives in IndexedDB ‚Äî does not pass through the HTTP cache / SW fetch handler |
| `navigator.serviceWorker` cookies / session | Browser manages these independently |

Workbox's `navigateFallback` handles SPA routing for HTML ‚Äî it serves `/index.html` from precache for all navigation requests that don't match a precached asset.

## 8. Offline Detection Layers

CEV uses two complementary mechanisms:

| Layer | Mechanism | When it fires |
|-------|-----------|---------------|
| Network events | `online` / `offline` DOM events via `initNetworkListeners()` | Immediately on connectivity change |
| SW message channel | SW posts `{ type: 'ce-bg-sync-replayed' }` to tabs after replay | When SW successfully resyncs after being offline |

`OfflineBanner.vue` reacts to `useSyncStore.online` (layer 1). No changes needed for PWA support ‚Äî the banner already works offline.
